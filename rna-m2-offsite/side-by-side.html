Sensitivity ?Controls how statistically significant a change must be to trigger an alert. Lower p-value = more confident the change is real (fewer false positives, but might miss subtle shifts). The default 0.05 means "95% confident this is a real change."

How confident should we be that the change is real before alerting?
Step change
A statistically significant step up or down in value distribution
Trend change
An overall trend change occurring at this point
Spike
A significant sudden increase at this point
Dip
A significant sudden decrease at this point
Distribution change
The overall distribution of values has changed significantly
No manual thresholds required. CHANGE_POINT analyzes the time series statistically and fires when it detects a significant shift.

Template integrity: can we re-open in template mode?

When a rule is created from a template, we store the template ID and a hash of the generated ES|QL. When re-opening, we compare the current query to the original to decide what's safe.

Scenario 1: Rule untouched since template creation
FROM metrics-system.cpu-default | STATS avg_value = AVG(system.cpu.total.norm.pct) BY host.name, ts = BUCKET(@timestamp, 5 minutes)
TEMPLATE MODE

ES|QL matches stored hash exactly. We can parse the AST, extract all template values, and open the full template form.

Scenario 2: User changed data source and metric field
FROM metrics-system.cpu-default
FROM metrics-system.memory-default
| STATS avg_value = AVG(system.cpu.total.norm.pct)
| STATS avg_value = AVG(system.memory.used.pct)
BY host.name,
ts = BUCKET(@timestamp, 5 minutes)
TEMPLATE MODE

Query structure matches the template's AST pattern. Only parameterizable values changed. We extract the new values and populate the template form.

Scenario 3: User added a WHERE filter
FROM metrics-system.cpu-default
| WHERE cloud.provider == "aws"
| STATS avg_value = AVG(system.cpu.total.norm.pct)
BY host.name,
ts = BUCKET(@timestamp, 5 minutes)
TEMPLATE MODE (partial)

Structure recognizable — template's core pattern is intact with a known extension (WHERE filter). Opens in template mode with the filter populated.

Scenario 4: User added significant new ES|QL stages
FROM metrics-system.cpu-default
| STATS avg_value = AVG(system.cpu.total.norm.pct)
BY host.name,
ts = BUCKET(@timestamp, 5 minutes)
| SORT host.name, ts
| CHANGE_POINT avg_value ON ts
| WHERE type IS NOT NULL AND pvalue < 0.05
EDITOR ONLY

Query has new pipeline stages that fundamentally change the logic. Template mode unavailable. Full editor with a note that template was detached.

How detection works

Each template defines an AST pattern — the structural shape of its generated ES|QL. When re-opening a rule:

  1. Hash check — if ES|QL hash matches stored original, open in template mode.
  2. Pattern match — parse AST, compare structure. If pipeline stages match with only parameter-level changes, extract values.
  3. Known extensions — some edits are recognized (WHERE filter, extra GROUP BY). These map to template fields.
  4. Structural divergence — new/removed stages or fundamentally different shape = editor only.

The rule stores: template_id, template_version, and query_hash. Template detachment is a one-way door for structural changes, but parameter edits are always recoverable.
