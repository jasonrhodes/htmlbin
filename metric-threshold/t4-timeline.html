The two queries that run at T4
Query 1
Evaluation query
Groups the data by service.name over both the current and previous windows, and returns groups that are either above the threshold now — or had data last window and none this window.
GET metrics-*/_search
{
  "aggs": {
    "services": {
      "composite": { "sources": [{ "service.name": ... }] },
      "aggs": {
        "currentPeriod": // T3 → T4
          { "filter": { "range": { "@timestamp": { "gte": "now-1m" } } },
            "aggs": { "avg_cpu": { "avg": { "field": "system.cpu" } } } },
        "lastPeriod": // T2 → T3
          { "filter": { "range": { "@timestamp": { "gte": "now-2m", "lt": "now-1m" } } } },

        "keep_group": {
          "bucket_selector": {
            "script": "currentPeriod.avg_cpu > 0.85
                     || (lastPeriod._count > 0 && currentPeriod._count < 1)"
          }
        }
      }
    }
  }
}
Returns
svc = A avg_cpu: null · currentPeriod._count: 0 no data
svc = B avg_cpu: 0.87 firing
svc = D avg_cpu: 0.89 firing
svc = C is absent — had docs this window (avg_cpu = 0.62), but failed both branches of keep_group. The framework auto-recovers its alert.
Query 2
Verification msearch
For each group that was flagged as no-data on the previous run, ask: are there any docs for this group right now — with the CPU threshold removed? Confirms the group is still genuinely missing, vs just currently below threshold.
// runs once per group carried in state.missingGroups
// (at T4, no groups carry over yet — this fires at T5 for svc=A)

GET metrics-*/_search
{
  "size": 0,
  "terminate_after": 1,
  "track_total_hits": true,
  "query": {
    "bool": { "filter": [
      { "range": { "@timestamp": { "gte": "now-1m" } } },
      { "match": { "service.name": "A" } }
      // ← no CPU threshold filter
    ]}
  }
}
Returns
svc = A hits.total.value: 0 still missing
If the msearch had returned > 0 hits, svc = A would be dropped from state.missingGroups — meaning it has docs again, just below the CPU threshold. The alert would then recover normally.

Query 1 runs every execution and catches new disappearances at the moment they happen. Query 2 runs on subsequent executions for each group still flagged as missing, distinguishing "truly gone" from "back but below threshold".

Inside Query 1 — base query run twice, condition applied at the end

In v2 we talk about a base query (the data, without the alert condition) and the alert condition (the threshold that decides firing). Query 1 is that same split, expressed in one ES search: the base query runs against both windows via filter sub-aggs, and CPU > 0.85 is only applied at the very end by the bucket_selector. That's what lets a single query distinguish a group that disappeared (svc=A) from a group that's just below threshold (svc=C).

1
Group by service.name with a composite agg
The rule groups by service.name. A composite aggregation paginates over every unique value — one bucket per service. (A plain terms agg caps out at the bucket limit; composite scales and supports after_key to walk the pages.) 
GET metrics-*/_search
{
  "size": 0,
  "aggs": {
    "services": {
      "composite": {
        "sources": [
          { "service.name": { "terms": { "field": "service.name" } } }
        ]
      }
    }
  }
}
After this: one bucket per distinct service that reported anywhere in the search range — svc=A, B, C, and D all get buckets. Nothing said yet about time windows or CPU.
2
Drill in: filter sub-aggs for currentPeriod and lastPeriod
Inside each service bucket, two filter sub-aggs slice the docs by @timestamp — currentPeriod (T3 → T4) and lastPeriod (T2 → T3). These are aggs-context filters: they scope sub-aggs to a subset without touching hits or the sibling bucket. 
"services": {
  "composite": { ... },
  "aggs": {
    "currentPeriod": // T3 → T4
      { "filter": { "range": { "@timestamp": { "gte": "now-1m" } } } },
    "lastPeriod":    // T2 → T3
      { "filter": { "range": { "@timestamp": { "gte": "now-2m", "lt": "now-1m" } } } }
  }
}

Drill further: avg_cpu as a sub-agg of currentPeriod
The metric — avg of system.cpu — is a sub-agg of currentPeriod, so it averages only docs in the current window. lastPeriod doesn't need a metric; the rule uses just its _count to detect disappearance. 
"currentPeriod": {
  "filter": { "range": { "@timestamp": { "gte": "now-1m" } } },
  "aggs": {

4
Tie it together — bucket_selector applies the alert condition
A bucket_selector pipeline agg runs after the sub-aggs resolve and prunes the composite buckets. Keep if the threshold fires or the group disappeared (had data last window, none now). Everything else drops out of the response — and the framework recovers those alerts. 
"keep_group": {
  "bucket_selector": {
    "script": "currentPeriod.avg_cpu > 0.85                            // alert condition
             || (lastPeriod._count > 0 && currentPeriod._count < 1)"  // disappeared
  }
}
