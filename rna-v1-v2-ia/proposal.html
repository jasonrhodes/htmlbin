v1 + v2 Coexistence — Navigation & IA Proposal

A design exploration for how v1 and v2 alerting coexist during the M2 release window and beyond.

ℹ

Context: The April offsite locked in the split decision — rules are split, alerts are split. Combined/interleaved was ruled out. Now we need to define how the split feels in the product, not whether to split.

⚠

Current state: v2 shows up in Management as "V2 Alerting Preview" — a completely separate section, no cross-links to v1, no migration path visible, a name that says "experimental" to users. This is a placeholder, not a product.

The five problems to solve

#ProblemStatusThis document
1 Naming — can't say "v1/v2" or "legacy" in UI. Gating downstream UI work. Urgent Blocking Tab 1 → Naming
2 Navigation IA — where do both systems live, how does discovery work. Open Tab 2 → Navigation IA
3 Migration UX — copy/clone model needs to be easy, safe, compelling. Open Tab 3 → Migration
4 Mixed-state alerts — many customers will run v1+v2 simultaneously for months/years. Open Tab 4 → Mixed-state
5 v1 consolidation (#25) — Stack and O11y have separate v1 alert lists. Needed before mixed-state is addressable. Empty issue Tab 5 → #25

Key constraints (locked decisions)

Architecture

v2 runs alongside v1. No forced migration. Users adopt v2 at their own pace. v1 is not going away.

UX (April offsite ADR)

Split experience. Separate rule lists, separate alert lists. No combined/interleaved list — technical and semantic cost is too high.

Naming

Can't use "v1/v2" in the UI. Can't use "legacy" (value judgment). Natasha circulated a candidates doc.

Tab 1 — Naming

What to call v1 and v2 in the UI, docs, and in-app copy. This is the blocking problem.

⚠

The core tension: The names need to be distinct enough that users never confuse them, but neither should imply the other is worse. "Legacy" fails that bar. "v2" exposes internal versioning. Something in between is needed.

Natasha's candidates (summarized)

Three concrete naming options

Option A
v1 Rules & Connectors (no change)
v2 Alerting
Clean, forward-looking. v2 owns the term "Alerting".
Both systems have a "Rules" object inside them — cross-context confusion possible.
Option B
v1 Rules & Connectors (no change)
v2 Alerting — section renamed from "V2 Alerting Preview"
Disambiguation when needed: v1 items = "rule", v2 items = "alert rule" or "episode".
Minimal change. v1 keeps its current identity. v2 drops the "Preview" label.
Users may not realize the two sections serve the same job until they read copy.
Option C
v1 Classic Rules
v2 Alerting
Unambiguous. "Classic" signals the relationship without "legacy".
"Classic" still carries an implicit value judgment — may feel dismissive to v1 teams.

Recommendation: Option B — minimum disruption, clean forward name

The core idea: v2 gets the clean name "Alerting". v1 keeps its existing name "Rules & Connectors" — no rename required, no value judgment. Disambiguation is handled by context (which section you're in) and by the specific object names (rule vs. alert rule vs. episode).

Why "Alerting" works for v2: The v2 product genuinely is the alerting system going forward. It subsumes detection, lifecycle, notification, and workflow — the full alerting arc. Naming it "Alerting" is accurate, not aspirational. v1 keeps "Rules & Connectors" which accurately describes its components (you configure rules and connectors, you don't configure alerting).

Tab 3 — Migration UX

The copy/clone model for moving a rule from v1 to v2. Must be easy, safe, and reversible.

ℹ

Principle: Migration is never automatic. Users initiate each rule migration on their own timeline. v1 rule stays active until the user explicitly disables or deletes it. The model is copy → verify → disable — not replace.

Tab 4 — Mixed-state Alerts

Customers running v1+v2 simultaneously will need a coherent view of their alerts across both systems.

⚠

The problem: The split decision means separate lists. But an SRE on call doesn't care which alerting system fired — they need to see everything relevant to their service, regardless of origin. The split should be an implementation detail, not an on-call UX.

Where v1 and v2 alerts live (data layer)

.alerts-observability.*Observability rules

ℹ

Key structural difference: v1 alerts are updated-in-place — only the current state survives. v2 rule events are append-only — full history is queryable. A unified list that combines both would need to read from fundamentally different document shapes and present them coherently. This is hard, but Discover makes it tractable.

Three approaches for mixed-state visibility

Approach A — Discover as the unifying surface Recommended for M2

Power users can query both alert indices with ES|QL from a single Discover session. Both v1 and v2 alerts are queryable data. No new UI component needed — just the right Discover data view and a starter query.

Query both alert systems from Discover:

// v2 episodes currently active FROM .rule-events | WHERE episode.status == "active" AND @timestamp > NOW() - 24h | STATS count = COUNT() BY rule.name | SORT count DESC
// v1 alerts currently active (from .internal.alerts-*) FROM .internal.alerts-* | WHERE kibana.alert.status == "active" AND @timestamp > NOW() - 24h | STATS count = COUNT() BY kibana.alert.rule.name | SORT count DESC

Ship a "Mixed-state Alerting" Discover session template as a saved search, accessible from a cross-link in both the v1 alerts page and the v2 Episodes list. No new page, no new API.

✓ Zero new UI components ✓ Fully queryable / customizable △ Requires ES|QL literacy
Approach B — Cross-links between v1 and v2 alert pages

Add contextual cross-links on the v1 alerts page and the v2 Episodes list so users can switch context. Lightweight: just links, no data merging.

↗
v1 Alerts → v2
Also running v2 rules? View episodes in Alerting →

Shown on the v1 Stack Alerts page. Only visible when at least one v2 rule exists in the space.

↙
v2 Episodes → v1
Still running v1 rules? View alerts in Rules & Connectors →

Shown on v2 Episodes list when v1 rules are active in the same space.

Approach C — Unified "All Alerts" page (M3+)

A purpose-built cross-system alerts page that normalizes v1 and v2 alerts into a shared schema, built for on-call triage use. Separate list UI that reads from both index families, presents a common surface with filtering by system (v1 | v2 | both).

✓ Best UX for mixed-state ops teams △ High build cost △ Schema normalization is hard → M3 candidate

M2 recommendation: Approaches A + B in combination

✓

Ship cross-links (B) immediately — zero new components, just conditional banners. Offer the Discover-based unified query (A) as a saved search template, linked from both alert pages. This covers 80% of the mixed-state need with minimal cost. Plan Approach C for M3 once the v1 consolidation (#25) makes the v1 data more accessible.

Tab 5 — v1 Alerts Consolidation (#25)

Issue #25 (child of #331) proposes consolidating Stack and O11y v1 alert lists. Currently an empty issue body.

⚠

Why this matters for v1/v2: The mixed-state Approach C (unified alert page) becomes easier once the v1 alert surface is consolidated. If we build a cross-system page while v1 has multiple entry points, we're unifying 3 things (Stack alerts + O11y alerts + v2 episodes) rather than 2. Doing v1 consolidation first reduces scope.

The consolidation target (#25): Stack + O11y into one. Security Alerts stays separate — its alert model (SIEM detections) is too distinct. The goal: one v1 alert list in the Stack context that includes both Stack-level rules and Observability rules. O11y's solution-specific alerts page may remain as a deep-link from the O11y solution context, but the canonical "see all alerts" surface should be the Stack page.

Issue #25 body — proposed content

Proposed issue body for #25

Goal

Consolidate Stack and Observability v1 alert lists into a single alerts page accessible from Management. This is a prerequisite for the cross-v1/v2 mixed-state alert experience (#331).

Acceptance criteria

  • A single alerts list in /management/insightsAndAlerting/triggersActions/alerts (or equivalent v2 path) shows alerts from both Stack-level rules and Observability rules.
  • Filtering by rule type / solution is available (so O11y engineers can still focus on O11y rules).
  • O11y solution-level /observability/alerts can remain as a convenience deep-link but should derive from the same data source as the consolidated page — not a separate fetch.
  • Security alerts (/security/alerts) are out of scope for this issue.
  • The consolidated page is used as the link target from v2 cross-links (Tab 4, Approach B).

Dependencies

  • Requires coordination between triggers_actions_ui (Stack) and observability plugin teams.
  • RBAC: consolidated page must respect solution-specific alert visibility rules.

Out of scope

  • v2 Episodes — v2 alerts are on a separate list by decision (split ADR). Cross-system mixed-state is tracked in #331 parent.
  • Security Alerts page consolidation.

Sequencing relative to v2 work

Can proceed in parallel with v2 work
  • Naming change (Tab 1) — purely v2-side rename, no v1 dependency
  • Navigation IA change (Tab 2) — v2 section rename, section ordering
  • Cross-links between v1 and v2 (Tab 4, Approach B)
  • Migration entry point on v1 rules list (Tab 3)
Blocked on or benefits from #25
  • Cross-system "All Alerts" unified page (Tab 4, Approach C)
  • The Discover saved search template (needs to link to the consolidated page)
  • Migration verification UX — "compare v1 vs v2 firings" (Tab 3, Step 5)
