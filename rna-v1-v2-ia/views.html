View 1 — Alerting Hub page

The landing page is a status overview, not a list. Users navigate from the hub to whichever view they need. The two alerting systems get distinct cards; the "Active Now" mixed view is a third entry point.

Core idea: When a user clicks "Alerts" in the O11y nav, they land on a hub page that shows counts + health for both systems, not an immediate list. "What's happening?" is answered at a glance; clicking through gets you the list you need. This avoids forcing a tab choice on users who just want the summary.

View 2 — Active Now (mixed)

A minimal unified list of everything currently firing. Both systems. Deliberately limited columns. The left colored bar is the system indicator — no separate column needed.

Core idea: Show the ACTIVE things, not the whole history. Snooze works on both from this view. System-specific actions (ack for TS episodes, rule navigation for Snapshot) live in the flyout that opens when you click a row. The left-border color pattern is borrowed from calendar apps — a lightweight, learned visual shorthand.

View 3 — Entity-centric alerts

Organize by what's being monitored (service, host, cluster), not by which alerting system fired. Both v1 and v2 alerts about the same entity appear together. The system is secondary.

Core idea: SREs don't ask "what's in my v2 alerting system?" — they ask "what's wrong with checkoutservice?" This view is organized around entities. Within each entity card, alerts from both systems appear together with system indicators. This is the most outside-Kibana-boxes idea in this prototype.

The big unlock: If two rules (one v1, one v2) are both monitoring checkoutservice's error rate, an SRE sees both on one card. This is the "creative solution for mixed state" from the offsite — side-by-side by entity, not by system. Works because grouping fields (service.name, host.name) are shared between v1 and v2 even though schemas differ.

⚠

Limitation: Entity attribution requires that both v1 and v2 alerts can be mapped to the same entity key (service name, host name, etc.). v2 uses the grouping fields from the ES|QL query. v1 needs the alert to carry compatible fields. This works for APM/Infra rules today; may not work for all v1 rule types.

View 4 — Lifecycle board

A Kanban-style board organized by alert lifecycle state. v1 alerts only appear in "Active" and "Resolved". v2 episodes appear in all columns. The board makes lifecycle gaps between the systems instantly visible.

Core idea: Lifecycle IS the primary organizing principle for v2. v1 doesn't have Pending or Recovering — those columns are v2-only. The board makes this structural difference self-evident without explanation. Users naturally understand: "if it's in the Active column, it's from either system; if it's in Pending, it's always v2."

View 5 — Rules: featured ES|QL section

v2 rules as a "featured" rule type — not a separate tab. Sections within one page. ES|QL rules are surfaced at the top with distinct styling. Below them, all v1 rule types in the existing list experience.

Core idea: Jason mentioned framing v2 as a "first-class ES|QL rule type" — the best one, the only one you'll ever need. One way to express that in the UI: feature it at the top of the rules list, not on a separate tab. The list is one list; the section header creates the visual separation. Drawback: the tables genuinely have different columns, which is why tabs may still be the right answer. This is the alternative to explore.

Snooze — the critical shared action

Snooze operates on different things in v1 vs v2: v1 snoozes the rule itself (all alerts from that rule). v2 snoozes a specific series/episode. The UX of snooze should reflect this: from the mixed "Active Now" view, snoozing a v1 item snoozes the rule, snoozing a v2 item snoozes the episode.
Snooze confirmation modal
Snooze "CPU > 85%" SNAPSHOT
This will snooze all alerts from this rule for the selected duration. Applies to all 2 currently active alerts.
Snooze "Service Health Monitor · checkoutservice" TIME SERIES
This will snooze this episode (checkoutservice) only. Other episodes from this rule will continue to fire.

The flyout pattern is how you handle divergent actions without cluttering the list. The list only shows the shared actions. System-specific actions live one click deeper. Users learn the pattern quickly: everything shared is at the top, system-specific stuff is in the lower section with a colored header matching the system badge. The system badge color (amber/blue) is the consistent visual anchor across all views.
