v1 + v2 Coexistence Proposal
DRAFT v2 · May 2026

Tab 0 — Corrected mental model

What v1 and v2 actually are, what they share, and where they genuinely differ. Starting over from here.

⚠

What the first prototype got wrong: Framed v1 as "Rules & Connectors" and v2 as "Alerting" — implying only v2 does alerting. In reality both are complete alerting systems with rules, alerts, and connectors. The naming problem is harder because of this.

What both systems share

Rules — configured detection logic that runs on a schedule and produces alerts when conditions are met.

Alerts — the output when a rule's condition is met. Both systems produce something called an alert.

Connectors / Actions — how alerts turn into notifications (Slack, email, PagerDuty, etc.).

Where they genuinely differ

Dimensionv1v2
Rule model Catalog of specialized rule types (Metric Threshold, Log Threshold, APM Latency, ML Anomaly, ES Query with 3 modes including ES|QL…). You pick a type and fill in parameters. One model: any ES|QL query. You write the detection logic. No "type" picker. Note: v1's ES Query has an ES|QL mode, so ES|QL alone doesn't distinguish them.
Alert storage Single mutable document per group, updated in place on every execution. Only current state survives. Append-only rule event document written per execution. Full history queryable. No overwriting.
Lifecycle tracking Per-rule-type, opaque to the user. Each rule type implements its own lifecycle. Explicit episode state (Inactive → Pending → Active → Recovering → Inactive). Configurable thresholds for each transition. Not all v2 rules use this — signal-kind rules write events without episode tracking.
Data on alert docs Each rule type decides what fields land on the alert document. If you want different fields, you can't change it. User controls via ES|QL KEEP. Every field the query computes lands in data.* on the rule event. User decides what's stored.
Recovery condition Per-rule-type, fixed. User-specified: no_breach (group disappears from results) or a custom recovery query with its own threshold.
Notification routing Actions configured per-rule. Each rule has its own action (connector + frequency). Action policies are global — one policy can match episodes from many rules simultaneously. Decoupled from rules.
Rules on rules Not possible. Natural consequence of append-only events: query .rule-events in a second rule to create higher-order detections.

Scope of this proposal

ℹ

Focused on Observability. Rules are global (→ /app/rules). Episodes/alerts live in O11y for now. Security is out of scope.

Two surfaces to redesign: (1) /app/rules — extend with a second tab for v2 rules; (2) /app/observability/alerts — extend with a second tab for v2 episodes.

⚠

Naming is unresolved and hard. Both systems do alerting. Both have rules. ES|QL doesn't cleanly distinguish them. Tab 4 presents the naming problem honestly with options — not a recommendation yet.

Tab 4 — Naming

The tab labels for v2 content on /app/rules and /app/observability/alerts. No good answer yet — this is a brainstorm, not a proposal.

⚠

Why naming is hard here: Both systems do alerting. Both have rules. ES|QL alone doesn't distinguish them (v1 ES Query has an ES|QL mode). "Custom" implies v2 is the exception. "Classic" implies v1 is outdated. "New" is temporary. Any version-based name ("v2") is internal jargon. Whatever we pick needs to work in two contexts: the Rules tab label AND the Alerts tab label.

alerts tab: "Alerts"

Honest. Communicates "this is the new thing" without committing to a final name during preview.

The deeper problem: consistent naming across contexts

💡

The tab label is just the surface. The real naming question is: what do we call the whole v2 system in a sentence? "Go to [?] to see your v2 rules." "Your v1 alerts are in Alerts. Your [?] are in [?]." Whatever word we pick for the tabs also has to work in breadcrumbs, docs, error messages, and the migration UX. The tab label is the tip of the iceberg.

Natasha's candidates (from her naming doc): The leading option was "Rule-based Monitoring and Alerting" but concern was it implies v1 and v2 are complementary. Other candidates: Condition Monitoring, Lifecycle Alerting, Stateful Alerting, Adaptive Alerting, Rules and Signals. None of these directly address the tab-label context — they're section-level names, not distinction-within-a-page names.

One angle not in Natasha's list: lean into the output difference — v1 produces "alerts" (current state), v2 produces "alert events" (history). Tab names: "Alerts" vs "Alert Events" or "Alert History". This emphasizes the append-only/queryable aspect which is a genuine user benefit.
